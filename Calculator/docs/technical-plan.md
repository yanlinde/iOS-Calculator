# Calculator 技术方案

## 1. 技术选型

### 推荐方案
- 客户端：SwiftUI
- 语言：Swift
- 最低系统版本：iOS 17+
- 数据持久化：SwiftData
- 架构：MVVM
- 测试：XCTest

### 选择理由
- 目标平台是中国区 App Store 上架的 iPhone App，原生 iOS 方案最稳妥。
- 当前产品是单页标准计算器，交互和动画都比较轻量，SwiftUI 足够覆盖。
- SwiftData 适合本地历史记录持久化，复杂度低，后续也容易扩展。
- MVVM 便于把计算逻辑、显示逻辑和界面布局分开，方便测试和迭代。

### 不推荐作为 V1 首选的方案
- Flutter：跨端能力强，但当前没有 Android 诉求，增加技术面。
- React Native：开发快，但对原生上架与长期维护不如纯原生收敛。
- UIKit：可行，但对当前单页产品来说开发成本高于 SwiftUI。

## 2. 产品实现边界

### V1 只做
- iPhone 竖屏
- 单页面计算器
- 严格数学优先级计算
- 本地持久化历史记录
- 历史最多 10 条
- 历史区可上下滚动

### V1 不做
- iPad
- 横屏
- 科学计算
- 清空历史
- 云同步
- 设置页

## 4. 页面结构设计

### 主页面布局
- 顶部安全区占位
- 历史记录滚动区域
- 上方小字结果区
- 下方大字主显示区
- 5 行按键区

### 历史记录区规则
- 固定可见高度，对超出内容启用纵向滚动。
- 按时间正序展示，最新一条在最下方。
- 最多保存 10 条；超过 10 条时，删除最旧一条。

### 双层结果区规则
- 小字结果区：
  - 按等号前显示当前表达式截至此刻的计算结果
  - 按等号后隐藏
- 大字主显示区：
  - 按等号前显示实时输入内容
  - 按等号后显示最终计算结果

## 5. 计算引擎设计

### 输入模型
- 将用户输入转为 Token 序列
- Token 类型：
  - 数字
  - 小数点
  - 运算符
  - 百分比
  - 一元负号

### 计算规则
- 必须严格遵守数学优先级
- 乘除优先于加减
- 不支持括号输入
- `+/-` 作用于当前编辑中的数字
- `%` 默认作用于当前数字，将其转为原值除以 100

### 推荐实现
- 使用“中缀表达式 -> 后缀表达式 -> 求值”的标准方案
- 可采用 Shunting-yard 算法处理优先级
- 计算结果统一用 `Decimal`，避免 `Double` 浮点误差直接暴露

### 实时预览结果
- 当表达式处于可计算状态时，实时计算并展示在小字结果区
- 当末尾是运算符时，忽略未完成项的结尾状态，按可解析部分计算
- 若当前表达式非法，则小字结果区不显示或维持上一次有效结果

### 异常处理
- 除数为 0：展示 `Error`
- 非法表达式：不崩溃，不写入历史
- 超长结果：使用格式化与缩放策略

## 6. 状态管理设计

### CalculatorViewModel 维护的核心状态
- `inputExpression`
- `displayExpression`
- `previewResult`
- `finalResult`
- `isShowingFinalResult`
- `historyItems`
- `errorState`

### 事件类型
- 点击数字
- 点击小数点
- 点击运算符
- 点击等号
- 点击清空
- 点击删除
- 点击百分比
- 点击正负切换

### 状态切换原则
- 任何一次用户输入后，界面状态都从单一 ViewModel 派生。
- 历史记录更新只在成功按下等号且结果合法时触发。
- 错误状态只能由新的合法输入或 `C` 清除。

## 7. 数据模型设计

### 历史记录模型
```swift
@Model
final class CalculationHistoryItem {
    var id: UUID
    var expression: String
    var result: String
    var createdAt: Date
}
```

### 数据规则
- 最多保留 10 条
- 新记录追加到尾部
- UI 按数组顺序展示，保证最新一条在最下方

## 8. 显示格式化方案

### 数字展示
- 千分位分隔
- 尽量去掉无意义末尾零
- 保留用户输入中的必要小数结构

### 表达式展示
- 运算符显示为 `+ - × ÷`
- 内部可使用标准枚举或符号映射

### 长内容处理
- 历史记录单行右对齐
- 大字结果区优先缩放字号，再考虑裁切
- 不允许换行破坏主视觉

## 9. UI 实现建议

### 推荐页面结构
- `CalculatorScreen`
- `HistoryListView`
- `PreviewResultView`
- `MainDisplayView`
- `KeypadView`
- `CalculatorButtonView`

### 按键布局
- 5 行 x 4 列固定布局
- 按钮尺寸尽量按设计稿比例实现
- 功能键、数字键、运算键使用统一组件，不同样式配置区分

### 视觉令牌
- 背景黑色
- 数字键深灰
- 功能键浅灰
- 运算键橙色
- 文本分层为白色和浅灰

## 10. 测试方案

### 单元测试重点
- 优先级计算正确
- 百分比处理正确
- 正负切换正确
- 删除逻辑正确
- 异常表达式不崩溃
- 历史记录最多 10 条
- 新记录追加到尾部

### UI 测试重点
- 竖屏主流机型布局正常
- 双层结果区行为符合 PRD
- 历史区可滚动
- 等号后小字结果区隐藏

### 关键用例
- `3+2-5*0 = 5`
- `12+3*4 = 24`
- `100% = 1`
- `5+/- = -5`
- `1÷0 = Error`
- 连续写入 12 条历史后只保留最新 10 条

## 12. 开发顺序建议
- 第 1 步：搭建 SwiftUI 工程和主题骨架
- 第 2 步：完成计算引擎与单元测试
- 第 3 步：完成主页面静态布局
- 第 4 步：接通 ViewModel 与按钮事件
- 第 5 步：接入历史记录持久化
- 第 6 步：完成显示格式化与异常处理
- 第 7 步：完成 UI 测试与真机适配

## 13. 风险与规避

### 风险 1：实时预览结果与最终结果规则混淆
- 规避：先独立定义显示状态机，再接 UI。

### 风险 2：浮点误差导致结果显示异常
- 规避：核心计算使用 `Decimal`，不要直接使用 `Double`。

### 风险 3：长表达式挤压布局
- 规避：显示层做字号缩放和右对齐裁切。

### 风险 4：历史排序与滚动体验不符合预期
- 规避：明确“最新一条在最下方”，并为此写 UI 测试。
